# 创建听评课邀请 - 完整 Workflow

**CRITICAL — 处理"创建听评课邀请"意图时 MUST 先读本文件。**

本文件教 Claude 如何**编排多步查询 + 创建**,处理用户口语化的建课请求(如"给张三老师明天第三节建听评课")。

**开始前 MUST 先读 [`../../hr-shared/SKILL.md`](../../hr-shared/SKILL.md) + [`../SKILL.md`](../SKILL.md) + [`hr-invite-create.md`](hr-invite-create.md)**。

## 核心理念

**用户表达是渐进式、不完整的。Claude 的目标是把沟通轮次压到最少:**

1. 用户开口已说的信息 **不重复问**
2. 用户能一次给齐的信息 **合并成单次提示**
3. 只有真正需要用户决策的分叉(同名选谁/课型选哪个/节次歧义)才追问,**且一次问完**

**禁止**:把 API 依赖链(查教师依赖校区 ID)硬塞给用户,让用户按 Step 1→2→...逐轮回答。用户认知里这些信息是一坨的,该让 Claude 内部跑依赖链。

## 信息分类表(Claude 决策依据)

| 信息 | 来源 | 必填 | Round 1 问? | 说明 |
|------|------|------|------------|------|
| schoolId / campusId | config.json 自动注入 | ✅ | **不问** | `saas +login` 已写入 `~/.hr-cli/config.json`,shortcut/service 命令自动复用。**仅在 config 缺失时才引导用户登录** |
| teacherName | 用户提供 | ✅ | 缺就问 | 用户说了就不问 |
| 时间(日期 + 节次) | 用户提供 | ✅ | 缺就问 | 如"明天第三节" |
| 课型 courseTag | 用户给名字 or Claude 列选 | ✅ | ⚠️ 标可选 | 用户不说 → Claude 查完 Round 2 列选 |
| 班级 | 用户提供 | ✅ | 缺就问 | |
| 地址 | 用户提供 | ✅ | 缺就问 | |
| 邀请成员姓名 | 用户提供 | ✅ | 缺就问 | Claude 后台查 staffId,用户不感知 |
| **课程名称 `--name`** | **最终确认时问** | ✅ | ❌ Round 3 问 | 推迟到确认轮次 |
| teacherId / teachGroupId | Claude 查 | - | - | 查教师接口 |
| commentPaperId | Claude 查 | - | - | 查量表接口,默认第一个 |
| node 数值 | Claude 映射 | - | - | "第三节"→3 |

## 主流程(≤ 4 轮对话)

```
用户开口(任何变体表达)
    ↓
Round 1: 智能补齐提示(只问缺失项)
    解析用户已表达的信息,把【缺失的必填项】一次性列给用户
    课型标"不确定可不填",课程名称不问
    ↓ (用户回复)
Claude 内部:并行查询(用户不感知)
    ├─ groupmanage +search-teacher 查教师 → teacherId + teachGroupId
    ├─ course coursetag_get_lists 查课型(用户没给名字时)
    ├─ course coursepaper_get_suit_lists 查推荐量表 → commentPaperId
    └─ course course_node_get_lists 校验节次 → node 数值
    ↓
分叉判断:
    ├─ 无分叉 → 跳过 Round 2,直接进 Round 3
    └─ 有分叉 → Round 2 合并追问(见下)
    ↓
Round 3: 最终确认 + 问课程名称(原 Step 8.5)
    把完整信息列给用户确认,只展示人类可读字段
    追加一行问课程名称
    ↓
用户"确认" + 给课程名称 → +create 创建
    ↓
告诉用户邀请码
```

### Round 1 智能补齐提示(动态,不抛固定清单)

**不是固定模板**——Claude 先解析用户开口已表达的信息,**只列出用户还没给的必填项**。

**示例 A**(用户只说了"给张三老师建课"):
```
好的,给张三老师建听评课。还需要您补充:

1. **授课时间**:日期 + 节次(如"明天第三节"、"6/30 下午第二节")
2. **课型**:如"语文""数学"(不确定可不填,我列出可选的让您选)
3. **班级**:如"高一(3)班"
4. **地址**:如"A301"
5. **邀请评课人**:姓名,逗号分隔(如"李四, 王五")

学校/校区已从您的登录态自动读取,不必提供。
您可以一次回复,示例:
明天第三节, 语文, 高一(3)班, A301, 邀请李四和王五
```

**示例 B**(用户开口已说一部分:"给张三老师明天第三节建语文课"):
```
好的,给张三老师明天第三节建语文听评课。还需要补充:

1. **班级**(如"高一(3)班")和 **地址**(如"A301")
2. **邀请评课人**(姓名,逗号分隔)
```

**规则**:
- 用户开口已说的字段(老师/时间/课型)**绝对不在 Round 1 重复问**
- 课程名称不问,推迟到 Round 3
- 课型标注"不确定可不填",给用户留省略空间
- 邀请成员**只要姓名,不要提 staffId**(用户脑子里只有人名)

### Round 2 分叉合并(仅在需要时触发,一次问完)

并行查询跑完后,只有以下情况才进入 Round 2,且**所有分叉合并成一条消息**:

| 分叉类型 | 触发条件 | 处理 |
|---------|---------|------|
| 同名教师 | 多位**不同 staffId** 同名 | 列出让用户选 |
| 授课老师跨教研组 | 同一 staffId 出现在多个教研组 | 改问"本次授课属于哪个教研组"(说明:影响量表推荐) |
| 课型未定 | 用户没给课型 + 课型列表 > 1 个 | 列出让用户选 |
| 节次歧义 | "第三节"没说正课还是早读 | 让用户确认 |

- `+create` 只需要 member 的 `staffId`,不需要评课人的 teachGroupId
- 评课人的教研组对创建逻辑**零影响**,问了就是过度收集, 不需要问
- 只有**授课老师**的教研组才重要(决定量表推荐)
- 评课人通过 `+search-teacher` 查到 staffId 即可,即使返回多个教研组也忽略

**判断"同一人"**:比较返回结果里的 staffId 字段,相同就是同一人。同一 staffId 出现在多个教研组(跨学科任教)→ 不要问"哪位老师",改问"本次授课属于哪个教研组" (无需关注评课人的教研组)。

**展示规则**:`+search-teacher` 输出已自带友好区 + `[internal]` 区。Claude 转述给用户时**只展示友好区**。

**CRITICAL — 多个分叉必须合并成一条消息**:
```
✅ 正确(合并):
  找到 2 位张三:
    1. 张三 - 高中语文教研组
    2. 张三 - 小学语文教研组
  可选课型:1.公开课 2.精英课 3.听评课1
  请回复:老师序号 + 课型序号

❌ 错误(逐个问):
  "您是指哪位张三?" → 等回复 → "课型选哪个?"  ← 拆成两轮,禁止
```

**禁止**:有多个分叉时逐个问。

### Round 3 最终确认 + 问课程名称(原 Step 8.5 升级)

**CRITICAL — 所有参数齐后,Claude MUST 把完整信息列给用户确认,得到明确同意后才能 +create。**

**CRITICAL — 给用户看的清单 MUST 只用"用户能理解的字段",禁止暴露 ID 和技术术语。**

保留原 Step 8.5 全部字段翻译规则。**新增**:确认清单后追加一行问课程名称。

**字段翻译规则**(必须遵守):
| 后端字段 | 给用户看 |
|---------|---------|
| teacherId: 6968... | (不显示,只显示"张三") |
| courseTagId: 7196... | (不显示,只显示"语文") |
| node: 3 | 第三节(用 displayName,不用数字) |
| listenType: select | 指定人参与 |
| listenType: appointment | 按范围预约 |
| listenType: other | 其他 |
| commentPaperId: 1 | (不显示,只显示"通用教师评课表") |
| teachGroupId | (不显示,只显示"语文教研组") |
| schoolId / campusId | (不显示,只显示学校/校区名) |

**输出示例**:
```
请确认以下信息,回复"确认"后创建:

授课老师:   张三
教研组:     语文教研组
课型:       语文
授课时间:   2026-06-25 09:00:00
节次:       第三节
班级:       高一(3)班
地址:       A301
听课方式:   指定人参与
评课量表:   通用教师评课表
邀请成员:   李四, 王五(2 人)
课程名称:   (待定 —— 请输入,如"高一语文《荷塘月色》")
```

用户可能回复:
- "确认,课程名称 XXX" → 进 +create
- "把时间改到下午" → 修改后回到 Round 3 重新确认
- "取消" → 终止

### 创建(Round 3 通过后)

```bash
hr-cli course +create \
  --school-id X --campus-id Y --teacher-id T --teacher-name "张三" \
  --name "高一语文《荷塘月色》" --time "2026-06-25 09:00:00" --node 3 \
  --course-tag-id Z --class-name "高一(3)班" --addr "A301" \
  --teach-group-id G --current-user-teach-group-id G \
  --comment-paper-id P --comment-paper-name "通用教师评课表" \
  --member-ids "M1,M2"
```

返回 `courseInviteCode` → 告诉用户:"✅ 已创建,邀请码是 ABC123"

## 关键规则(保留)

### 规则 1:能查的不要问

```
❌ 错误:"commentPaperId 是多少?"
✅ 正确:调 coursepaper_get_suit_lists 自动拿推荐,默认第一个

❌ 错误:"node 是多少?(1-15)"
✅ 正确:用户说"第三节",Claude 自动转 node=3
```

### 规则 2:节次映射(易错)

```
用户口语              node 值
───────────────  ──────
"第一节"             1
"第二节"             2
...
"第十节"             10
"早读"               11
"早读第一节"          11
"晚自习第一节"        12
"晚自习第二节"        13
```

**不能瞎猜**——有歧义时进 Round 2 让用户确认。

### 规则 3:失败重试不要用相同参数

```
"当前课程已存在" → 改 --time 或 --node,不要重试相同参数
"campusId 不能为空" → 检查 --campus-id 是否漏传
```

## 示例对话

### 场景 A:用户开口已说一部分(常见,理想)

```
用户:给张三老师明天第三节建语文听评课

Claude:[解析已知:老师=张三,时间=明天第三节,课型=语文]
       好的,还需要补充:
       1. 班级?地址?
       2. 邀请谁评课?(姓名,逗号分隔)

用户:高一(3)班, A301, 邀请李四和王五

Claude:[并行查教师 → 唯一张三,查课型 → 确认语文,查量表,校验节次]
       [Round 3 最终确认 + 问课程名称]
       请确认:张三 / 明天第三节 / 语文 / 高一(3)班 / A301 / 邀请李四王五
       课程名称填什么?(如"高一语文《荷塘月色》")

用户:确认,课程名称"高一语文-荷塘月色"
```

**轮次:3 轮**(开口 → 补齐 → 确认)

### 场景 B:用户只说"建课"(最模糊)

```
用户:帮我建一节听评课

Claude:[Round 1 智能补齐,除 schoolId/campusId 外都缺]
       好的,创建听评课需要:
       1. 给哪位老师?(姓名)
       2. 授课时间(日期 + 节次)

       (学校/校区从登录态自动读取)

用户:张三,明天第三节,高一(3)班,A301,邀请李四

Claude:[并行查询,出现分叉:同名张三两位 + 用户没给课型]
       [Round 2 合并分叉]
       找到 2 位张三:
         1. 张三 - 高中语文教研组
         2. 张三 - 小学语文教研组
       可选课型:1.公开课 2.精英课 3.听评课1
       请回复:老师序号 + 课型序号

用户:老师1, 课型3

Claude:[Round 3 确认 + 问课程名称]
       ...

用户:确认,课程名称 XXX
Claude:✅ 已创建
```
**轮次:4 轮**(开口 → 补齐 → 分叉 → 确认)。分叉轮只在有歧义时出现。

## 相关命令速查

```bash
# 查教师
hr-cli groupmanage +search-teacher --school-id X --campus-id Y --name "张三"

# 查课型
hr-cli course coursetag_get_lists --data '{"schoolId":"X","campusId":"Y"}'

# 查推荐量表
hr-cli course coursepaper_get_suit_lists --data '{"schoolId":"X","campusId":"Y","courseTagId":"Z","teachGroupId":"T"}'

# 查节次
hr-cli course course_node_get_lists --data '{"schoolId":"X","campusId":"Y"}'

# 创建邀课
hr-cli course +create --school-id X --campus-id Y --teacher-id T ... --member-ids "M1,M2"
```
